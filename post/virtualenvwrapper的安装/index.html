virtualenvwrapper的安装

@ · Oct 30, 2015 · 2 min read

安装

支持的 SHELL

virtualenvwrapper 是在 Bourne Shell 兼容语法上定义的一个 Shell 函数的合集。在 OS X 和 Linux 系统上，可以用下面这些 Shell 进行自动化的测试：

  • bash
  • Ksh
  • zsh

它也许可已运行在其他的 Shell 上，如果你发现了它能够运行除了以上列出来的其它 Shell 上，请告诉我。如果你在没有完全重写它而让他运行在其他的 Shell 上，你可以通过bitbucket project page发送一个 Pull 请求。如果你写了一个克隆的版本去适应不兼容的 Shell 的话，也请告诉我，我会在这个界面上加上一个链接。

Windows 命令提示符

David Marble 已经将 virtualenvwrapper 到能够在微软的 Windows 命令提示符上运行的 Windows 批处理脚本。这是也是一个独立的分布式的再实现。你可以通过 PyPI 下载 virtualenvwrapper-win

MSYS

它可以使用 virtualenv 和本地的 Windows Python 安装文件封装MSYS。为了让它能够运行，你需要额外的定义一个环境变量，起名字为——MSYS_HOME,包括指向 MSYS 安装程序的根目录。


export WORKON_HOME=$HOME/.virtualenvs
export MSYS_HOME=/c/msys/1.0
source /usr/local/bin/virtualenvwrapper.sh

或者


export WORKON_HOME=$HOME/.virtualenvs
export MSYS_HOME=C:\msys\1.0
source /usr/local/bin/virtualenvwrapper.sh

根据你的 MSYS 安装程序，你可能需要在 MSYS_HOME/bin 文件中安装MSYS mktemp 二进制安装文件

Python 版本

virtualenvwrapper 是 Python 的 2.6-3.4 下进行测试。

基本安装

virtualenvwrapper 应该安装到 virtualenv 所在的全局的 site-packages 目录中。你需要管理员权限才能做到这一点。最简单的安装方式是使用 pip:


$ pip install virtualenvwrapper

或


$ sudo pip install virtualenvwrapper
警告

virtualenv 可以让你创建许多不同的 Python 虚拟环境。你应该只在你原始的 Python 安装文件上安装 virtualenv 和 virtualenvwrapper（i.e 不要在激活的虚拟环境中安装），这样所有的 pthon 环境才会依赖同一个版本。

另外一个将其安装在本地目录的全局的 site-packages 目录中的方法是（通常是 ~/.local）:


pip install --user virtualenvwrapper

Shell 启动文件

添加三行到你的 shell 启动文件中（.bashrc,.profile 等文件）来设置虚拟环境的位置，你的开发项目的位置，和这个安装包的脚本存在的位置。


export WORKON_HOME=$HOME/.virtualenvs
export PROJECT_HOME=$HOME/Devel
source /usr/local/bin/virtualenvwrapper.sh

在编辑完了之后，重新加载启动文件（e.g. 运行 source ~/.bashrc）

延迟加载

还有另一种延迟加载 virtualenvwrapper 初始化安装脚本的方法。直接用 virtualevnwrapper_laze.sh 来取代 virtualenvwrapper.sh .如果 virtualenvwrapper.sh 不存在你的\$PATH 中，设置 VIRTUALENVWRAPPER_SCRIPT 来指向它。


export WORKON_HOME=$HOME/.virtualenvs
export PROJECT_HOME=$HOME/Devel
export VIRTUALENVWRAPPER_SCRIPT=/usr/local/bin/virtualenvwrapper.sh
source /usr/local/bin/virtualenvwrapper_lazy.sh
提醒

使用延迟加载版本的启动脚本的时候，tab 部全功能在在第一条命令执行完之后才能使用。例如：在第一次 workon 之前是不能使用 tab 补全功能的。

快速启动

1.运行：workon 2.打印虚拟环境目录，或者为空 3.运行:mkvirtualenv temp 4.创建一个新的虚拟环境 temp 5.运行：workon 6.此时，temp 这个环境被包括进去了。

配置

virtualenvwrapper 可以通过改变环境变量来定制。 加载 virtualenvwrapper.sh 之前设置你的 shell 启动文件中的变量。

项目目录的位置

WORK_HOME 变量告诉 virtualenvwrapper 你的工作目录放在了哪里。默认值是\$HOME/.virtualenvs。如果这个路径不存在，那么当 virtualenvwrapper 加载的时候，就会创建这个目录。

项目链接文件名

PROJECT_HOME 变量告诉 virtualenvwrapper 你把项目的工作路径放在了哪里。在项目使用之前，这个变量必须被设置，路径必须被创建。

启动项目路径切换

VIRTUALENVWRAPPER_WORKON_CD 这个变量空指着当激活的时候，工作的项目目录是否会切换。默认值是 1，可以改变目录。如果设置成 0 的话，禁用了所有的 workon 调用行为。

Hook 日志的位置

VIRTUALENVWRAPPER_LOG_FILE 变量告诉 virtualenvwrapper，哪里可以为 Hook loader 写日志。默认是不写日志的。

Python 解释器，虚拟环境和环境路径

在启动过程中，virtualenvwrapper.sh 在$path中找到python和虚拟环境的程序，并提醒一会使用它们。这消除了当$path 改变时引起的冲突，并且在 virtualenvwrapper 没有安装或者不同版本的 virtualenv 被安装的虚拟环境上会从内部进行解释。由于这种行为，在使用 sourcing virtualenvwrapper.sh 是非常重要的。例如：


export PATH=/usr/local/bin:$PATH
source /usr/local/bin/virtualenvwrapper.sh

要覆盖\$PATH 搜索的值，为解释的完整路径设置 VIRTUALENVWRAPPER_PYTHON 的变量和 virtualenv 二进制文件的完整目录设置 VIRTUALENVWRAPPER_VIRTUALENV。这两个变量必须在 sourcing virtualenvwrapper 之前设置。例如：


export VIRTUALENVWRAPPER_PYTHON=/usr/local/bin/python
export VIRTUALENVWRAPPER_VIRTUALENV=/usr/local/bin/virtualenv
source /usr/local/bin/virtualenvwrapper.sh

virtualenv 的默认参数

如果定义这个应用的 VIRTUALENVWRAPPER_VIRTUALENV 需要一个参数，它们能在 VIRTUALENVWRAPPER_VIRTUALENV_ARGS 这里进行设置。同样的变量能被设置成默认的参数来传给 virtualenv.例如：设置成–no-site-packages 来保证所有的新环境是从site-packages目录中隔离出来的。

export VIRTUALENVWRAPPER_VIRTUALENV_ARGS='--no-site-packages'

临时文件

virtualenvwrapper 在 \$TMPDIR 中创建了临时文件。如果没有设置这个变量的时候，它使用 /tmp 这个路径。为了改变 virtualenvwrapper 的临时文件的路径，设置 VIRTUALENVWRAPPER_TMPDIR。

Side-Wide 配置

大多数的 Unix 系统有能够为所有的用户改变配置的能力。这通常有两种形式：编辑新的帐号的 skeleton 或者 编辑一个 shell 的全局的启动文件。

编辑新账户的 skeleton 文件，也就是说每一个新的用户会有它们私有的启动文件来预加载 virtualenvwrapper。我们可以通过注释或者删除那些语句所在的行来禁用这些。参考 Shell 和操作系统的文档来确定合适的文件去编辑。

修改一个 Shell 的全局启动文件的意味着这个 shell 的所有用户都会启动 virtualenvwrapper，它们不能禁用它。参考 Shell 和操作系统的文档来确定合适的文件去编辑。

更新到 2.9

版本 2.9 包括以前 virtualenvwrapper.project 分别提供的功能。 如果你有一个更老的版本安装项目的扩展，升级之前将其删除。

从 1.x 的升级

含包装功能的 shell 脚本已经更名为 2.x 系列，以反映炮弹比的 bash 其他的都支持的事实。 在你的启动文件，更改
